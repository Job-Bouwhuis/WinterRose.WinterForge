desired compilation steps for a compiler, from source text to binary output.
This system is designed to be modular, allowing for easy extension and modification of each stage in the compilation process.
However the system must have speed as a top priority, and memory usage as a secondary priority that 
can be sacrificed if it leads to significant speed improvements, simplification of the codebase, easier maintenance,
or better extensibility.

The system is at heart a configuration system, able to save and load the state of CLR objects.
The serialization part of the system comes simply from the fact that the system
can create instances of CLR objects, and set their fields and properties.
from there a simple serializer was built on top of that, which emits the necessary code to create the same objects
and set their fields and properties to the same values as the original objects.

for extra clarity, this system is not a general purpose serialization system, 
but rather a system that can serialize the state of CLR objects in a way that allows for the recreation of those objects in 
a different context, such as a different process or even a different machine.
with emphasis on speed while keeping memory usage as a secondary concern.
The features may make it seem like a serializer became scripting language, but thats not the idea.
the idea is that the system is used to create and configure objects.

In the end, me the developer are using it mostly as a serializer, ill be honest.
so yes, serialization is a big part of the system, but its not its main purpose.
its just a feature that was built on top.

The stages are as follows:

Source Text
    ↓
Lexer
    ↓
Parser
    ↓
Syntax Tree (AST)
    ↓
Binder / Semantic Model
    ↓
Bound Tree
    ↓
Analyzer Pipeline
    ↓
Lowering
    ↓
Opcode IR
    ↓
Optimizer
    ↓
Binary Emitter
